AgentGuard — Layer 2, 3, 4 Master Plan
=======================================

Context
-------
AgentGuard is an AI agent security middleware for a Microsoft hackathon. Layer 1 (Input Security) is complete. We now plan all remaining layers. This document covers:
1. Layer 2 (Output Security) — detailed implementation plan (building next)
2. Layer 3 (Pattern Detection) — initial design + Azure services
3. Layer 4 (Behavioral Monitoring) — initial design + Azure services
4. Full Azure services map — what we initially planned vs what Azure additionally provides


=======================================
Our Initial Design (What We Planned)
=======================================

Layer 2 — Output Security
  - Hallucination Detection    : Check if LLM output is grounded in source docs
  - PII Detection              : Detect and redact personal data in outputs
  - Output Toxicity            : Block harmful/toxic content in LLM responses

Layer 3 — Pattern Detection (from YAML)
  - Blocked patterns           : Match strings like "SYSTEM OVERRIDE", "curl", "BEGIN PRIVATE KEY"
  - Custom regex               : User-defined regex patterns
  - Block on match             : Configurable action

Layer 4 — Behavioral Monitoring (from YAML)
  - Tool call spike detection  : Detect unusual bursts of tool calls
  - Domain access monitoring   : Flag access to unexpected domains
  - New tool usage detection   : Alert when agent uses a tool for the first time
  - Data exfiltration patterns : Detect suspicious data flows
  - Tool firewall              : Per-tool allowlist/denylist (http_post, http_get, sql_query, file_system)
  - RBAC                       : Role-based tool access control (planner, executor, admin)
  - Human approval workflow    : Require approval for sensitive operations

=======================================

Module 1: Hallucination Detector (l2_output/hallucination_detector.py)
----------------------------------------------------------------------
Azure Service: Content Safety Groundedness REST API
API: POST {endpoint}/contentsafety/text:detectGroundedness?api-version=2024-09-15-preview
Credentials: Same CONTENT_SAFETY_ENDPOINT + CONTENT_SAFETY_KEY as L1 — no new Azure resource

Request format:
  {
    "domain": "Generic",
    "task": "QnA",
    "qna": { "query": "user question" },
    "text": "LLM output to check (max 7,500 chars)",
    "groundingSources": ["source documents (max 55,000 chars total)"],
    "reasoning": false
  }

Response format:
  {
    "ungroundedDetected": true,
    "ungroundedPercentage": 1.0,
    "ungroundedDetails": [{ "text": "the specific ungrounded text" }]
  }

Class: HallucinationDetector — follows PromptShields pattern (REST + requests)
Method: analyze(text, grounding_sources, query=None, confidence_threshold=0.9) -> ValidationResult
  - Auto-selects "QnA" when query provided, "Summarization" otherwise
  - Blocks when ungroundedDetected == True AND config says to block

Module 4: MELON Contrastive Injection Detector (l2_output/melon_detector.py)
----------------------------------------------------------------------------
Adapted from: MELON (ICML'25) — pi_detector.py reference implementation
Azure Service: Azure OpenAI Embeddings (text-embedding-3-large)
Credentials: Existing AZURE_OPENAI_ENDPOINT + AZURE_OPENAI_API_KEY (already in .env.example)

How MELON Works (Contrastive Detection):

  The core insight: if a tool output contains an injected instruction (e.g.,
  "send money to attacker"), the LLM will produce similar tool calls whether
  or not it sees the user's original query. MELON exploits this by running
  two passes:

  1. ORIGINAL RUN: LLM sees real conversation + tool outputs -> produces tool calls
  2. MASKED RUN:   LLM sees ONLY tool outputs (no user query/context) -> produces tool calls
  3. COMPARE:      Embed both sets of tool calls -> cosine similarity
     - HIGH similarity (>threshold) -> tool output is DRIVING actions -> INJECTION
     - LOW similarity -> user intent is driving actions -> SAFE

Our Azure-Native Adaptation:

  Instead of raw OpenAI client, we use Azure OpenAI SDK for embeddings.
  Instead of AgentDojo's pipeline, we fit into AgentGuard's ValidationResult pattern.

  Class: MELONDetector
  Constructor:
    def __init__(self, endpoint=None, api_key=None, deployment=None,
                 similarity_threshold=0.8, timeout_ms=10000):
        # Uses AZURE_OPENAI_ENDPOINT, AZURE_OPENAI_API_KEY, AZURE_OPENAI_EMBEDDING_DEPLOYMENT
        self.client = AzureOpenAI(azure_endpoint=endpoint, api_key=api_key, ...)
        self.similarity_threshold = similarity_threshold

  Method: analyze(original_tool_calls, masked_tool_calls) -> ValidationResult
    - Takes pre-computed tool call lists (Guardian orchestrates the two LLM runs)
    - Converts tool calls to text strings: "function_name(arg1=val1, arg2=val2)"
    - Embeds via Azure OpenAI text-embedding-3-large
    - Computes pairwise cosine similarity
    - If any pair > similarity_threshold -> injection detected

  Method: prepare_masked_context(messages) -> list
    - Extracts tool outputs from conversation, strips user query/intent
    - Builds the masked conversation (same approach as pi_detector.py lines 317-362)
    - Returns masked messages ready for a second LLM run

Key differences from original pi_detector.py:
  ORIGINAL (AgentDojo)                  | OUR ADAPTATION (AgentGuard)
  --------------------------------------|-------------------------------------------
  OpenAI client                         | AzureOpenAI client (hackathon)
  AgentDojo BasePipelineElement         | AgentGuard ValidationResult pattern
  Hardcoded text-embedding-3-large      | Configurable via YAML embedding_deployment
  Threshold 0.8 hardcoded               | YAML-configurable similarity_threshold
  print() debug statements              | logger.debug/info/warning
  Tight coupling to AgentDojo runtime   | Standalone module, Guardian orchestrates
  FunctionCall dataclass                | Generic dict-based tool call format

YAML config:
  output_security:
    melon_detector:
      enabled: false
      similarity_threshold: 0.8
      embedding_deployment: "text-embedding-3-large"
      block_on_detected_injection: true
      sensitive_functions:
        - "send_email"
        - "send_money"
        - "http_post"
        - "file_write"

Env vars needed: None new — reuses AZURE_OPENAI_ENDPOINT + AZURE_OPENAI_API_KEY

=======================================
Guardian Orchestration Flow
=======================================

LLM output -> @guard_output decorator (or Guardian.validate_output())
  -> HallucinationDetector.analyze()    # groundedness (if sources provided)
  -> PIIDetector.analyze()              # PII detection + redaction
  -> OutputToxicity.analyze()           # harmful content in output
  -> MELONDetector.analyze()            # contrastive injection detection (if tool calls present)
  -> If blocked + enforce -> raise OutputBlockedError
  -> If blocked + monitor -> log warning, allow through
  -> If dry-run -> skip all checks
  -> Return OutputValidationResult (includes redacted_text if PII found)

MELON-Specific Orchestration Detail:

  MELON requires two LLM passes — the Guardian must orchestrate this:

  Guardian.validate_output_with_melon(messages, tool_calls):
    1. original_tool_calls = extract tool calls from the LLM's actual response
    2. masked_context = MELONDetector.prepare_masked_context(messages)
       -> strips user query, keeps only system + tool outputs
       -> wraps tool outputs as "file contents" (following pi_detector.py pattern)
    3. masked_tool_calls = run LLM again with masked_context
       -> uses same LLM (Azure OpenAI) but with sanitized context
    4. result = MELONDetector.analyze(original_tool_calls, masked_tool_calls)
       -> embeds both via Azure OpenAI text-embedding-3-large
       -> cosine similarity comparison
       -> returns ValidationResult

  This means @guard_output for MELON needs access to the full conversation
  messages, not just the output text. The decorator will need an optional
  messages_param argument.
